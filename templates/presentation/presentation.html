If you're creating an application that requires user interaction, you have a few choices. You can make a command line tool, which is useful, but not very user friendly.

You can create a full blown GUI tool, which is very user friendly.

However, it might be difficult to create the same GUI experience on every platform your users might want to run it on.

Additionally, it can be difficult to keep a thick app updated on every user machine where it's deployed as the codebase advances.

You can create a web application.

Web applications have a couple of advantages over the previous two worth mentioning. The UI interface and application framework already exists on just about every user's platform; the web browser.

In addition, deploying and updating the application is as easy as refreshing the page.

I think there is a place for good looking, single page applications made with just HTML, CSS, and JavaScript. They are easy enough to create, and not so effort intense you'd think twice about replacing them with something more robust, like React, should the application become popular.

That's what this presentation is about, creating these kinds of applications.

As we go along we'll be doing two things, using CSS to style a simple web application, and using JavaScript to add interactivity to the application.

Applying CSS styles to a page really has two parts, knowing what styles to apply, and how to select an element from the page to apply it to. Right now we just want to add some minor styling to the entire page. In the Chrome debugger add the following to the home.css file: 

                
body {
    font-family: Roboto, Tahoma, sans-serif;
    font-size: 16px;
    background-color: floralwhite;
}

You should see the application page change immediately as you enter the above CSS styling. This is a pretty good way to develop CSS code as you can see the effects of your actions immediately.

In this case we've applied CSS styles to the body html tag. We've used body as the selector our CSS styles will be applied to when the browser renders the page. This also means all the children of the body tag will inherit it's styles.

Using HTML tags as selectors works fine for a lot of our styling needs, but often you'll need to select specific elements in a page. This is where the id and class attributes are useful.

The id attribute is used to identify a single tag within your HTML. For the most part I use id attributes to identify HTML elements that accept user input. Things like input fields, drop down select boxes, etc.

The class attribute is very useful, often moreso than id for two reasons:

  • It supports multiple values within the attribute and you can select on any of them.
  • The same value can exist in multiple class attributes, which makes it easy to select multiple HTML tags.

The asynchronous nature of JavaScript web application code can quickly become a mess. To help control that I'll present a little MVC framework that helps create separation of concerns. Conceptually this framework looks like this:

The Controller runs the show, and it has strong links to both the Model and View. It's primary job is to orchestrate the events happening in the application, like the user clicking buttons, entering data, etc.

When the Controller is created it is passed the existing Model and View instance variables so it can call their public interfaces directly.

The Model is responsible for all data interaction. Any data presented to the user comes from the model. Any data created or updated by the user goes through the model.

For the most part this means the Model is responsible for interacting with the API's provided by the server/service where the data originates and is persisted.

The View is responsible for updating the HTML DOM elements and changing what's displayed to the user. Any user interaction that changes the display is captured by the Controller, which then makes calls to the view to update the display accordingly.

In our application the server provides a simple HTTP API that transforms the user input data with some string and mathmatical operations, and returns the results as JSON data.

The Model accesses this API using the jQuery $.ajax(options) method, which is an asynchronous operation.

To handle this the $.ajax(options) call returns something called a Promise. This means the system promises to "call us back" when the operation is complete.

The Model returns the Promise to the Controller, which will act on either the success or failure of the action. This is the weak link between the Model and Controller.

The Controller will attach functions to the done and fail methods of the Promise.

For a done action the Controller will pass the data onto the View to update the display.

For a fail action the Controller simply outputs to the console the failure information.
